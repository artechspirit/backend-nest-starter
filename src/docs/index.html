Overview & Features

Starterkit ini dirancang menggunakan NestJS dan TypeScript dengan memprioritaskan keamanan (anti-DoS, proteksi CSRF), efisiensi performa (Redis caching, optimalisasi query DB), dan kepatuhan penuh terhadap standar tim.

🔒 Keamanan Kelas Produksi

Enkripsi Argon2id, cookie SameSite=Strict HttpOnly untuk refresh token, penanganan brute-force, dan mitigasi celah keamanan CPU DoS.

⚡ Redis-powered Caching

Mekanisme caching validation session JWT dan user permissions di Redis untuk memangkas latency database SQL secara masif.

📁 Stateless File Storage

Penanganan file upload ke Object Storage (S3/R2/GCS) secara streaming tanpa menulis ke lokal server disk. Dilengkapi validasi Magic Bytes.

🛠️ Arsitektur Bersih

Struktur modular, Dependency Injection ketat, DTO validation terenkapsulasi, dan transaksional DB ($transaction) terjamin.

Masalah & Solusi yang Diselesaikan Starterkit

Starterkit ini secara khusus menyelesaikan beberapa masalah teknis umum yang sering terjadi di tingkat produksi:

🔥 1. Kerentanan CPU DoS pada Verifikasi Refresh Token

Masalah: Cara lama melakukan hashing database session menarik seluruh session unrevoked dari DB ke RAM dan mengecek kecocokan Argon2 secara sekuensial dalam loop. Ini dapat membekukan event loop Node.js jika terdapat ribuan user.

Solusi: Refresh token kini diformat menjadi sessionId.tokenValue. Backend dapat mencari baris session secara instan menggunakan indeks kunci utama $O(1)$ di database SQL, lalu mengecek Argon2 hanya sekali pada session yang bersangkutan.

📊 2. Overhead Database SQL pada Validasi JWT

Masalah: Di setiap request ke endpoint terproteksi JWT, middleware menembak database untuk memverifikasi session aktif. Hal ini memperlambat API dan membebani server SQL.

Solusi: Kami mengimplementasikan Redis session caching. State keaktifan session disimpan di Redis (TTL 5 menit). Modul mutasi (seperti ganti password, logout, status update, soft-delete) akan secara otomatis membersihkan cache terkait agar session langsung ter-revoke.

🔑 3. DB Joins Berulang untuk Guard Hak Akses (RBAC)

Masalah: Setiap Guard mengecek hak akses user, backend melakukan JOIN tabel User, Role, dan Permission ke database.

Solusi: Izin user yang unik disimpan di Redis dengan TTL 1 jam. Saat admin memperbarui izin peran di modul roles, backend otomatis mencari user yang memegang peran tersebut dan meng-invalidasi cache izin mereka seketika.

💥 4. Connection Pool Leak & Timeout

Masalah: Prisma membuat connection pool dinamis yang berujung pada error `connection pool timeout` di bawah trafik tinggi.

Solusi: String koneksi di-tune secara eksplisit dengan connection_limit=10 dan pool_timeout=20, dipadukan dengan lifecycle cleanup NestJS OnModuleDestroy di PrismaService.

Quick Start

Ikuti langkah berikut untuk memasang dependensi dan menyalakan server lokal development:

Bash / Terminal 
# 1. Kloning dependensi pnpm
pnpm install

# 2. Setujui build dependensi untuk pnpm v10+ (jika diminta)
pnpm approve-builds

# 3. Salin environment template dan sesuaikan isinya
cp .env.example .env

# 4. Jalankan Docker Compose untuk menyalakan PostgreSQL, Redis, dan MinIO S3
docker compose up -d

# 5. Sinkronisasikan database schema dan jalankan database seeding
pnpm db:migrate
pnpm db:seed

# 6. Jalankan aplikasi dalam mode dev (watch mode)
pnpm start:dev

Architecture & Guidelines

Starterkit ini mematuhi standar arsitektur modular NestJS secara ketat:

1. Struktur Folder Modular

Setiap fitur/domain bisnis dikelompokkan dalam satu folder modul mandiri yang berisi Controller, Service, DTO, dan Spec File secara kohesif:

Struktur Direktori 
src/
├── cache/            # Caching global & redis config
├── common/           # Filter, Guard, Interceptor, Pipe global
├── config/           # Strongly-typed configuration & Zod schema validation
├── database/         # PrismaService & database modules
├── modules/          # Fitur Modul (Auth, Users, Roles, Upload)
│   ├── auth/
│   │   ├── dto/
│   │   ├── auth.controller.ts
│   │   └── auth.service.ts
│   └── users/
└── main.ts           # Bootstrapping & global middleware settings

2. Aturan Penting Developer (AI Guardrails)

Topik Pedoman Wajib
Environment Variables Dilarang menggunakan process.env secara langsung. Gunakan ConfigService dan validasi tipe via Zod saat startup.
Database Mutations Gunakan Prisma $transaction untuk penulisan berganda (multi-write) yang saling berhubungan demi menjamin integritas data (ACID).
Upload Files Selalu upload ke Object Storage secara stateless. Gunakan validasi Magic Bytes (file-type) untuk memeriksa tipe MIME asli.
Route Parameters Wajib memvalidasi tipe parameter route menggunakan pipe validator seperti ParseUUIDPipe.

3. Alur Detail Validasi JWT & Session Security Flow

Setiap request ke endpoint terproteksi akan melalui alur pengecekan keamanan berikut:

  1. Client Request mengirimkan Access Token (JWT) di HTTP Header `Authorization: Bearer `.
  2. JwtAuthGuard memotong token dan memverifikasinya secara kriptografi menggunakan kunci rahasia (`JWT_ACCESS_SECRET`).
  3. Setelah token valid, Guard memeriksa cache Redis untuk memastikan sesi masih aktif (tidak diblokir atau di-revoke). Jika cache hit dan user berstatus `ACTIVE`, akses langsung diberikan.
  4. Jika cache miss (data kadaluwarsa di Redis), Guard melakukan fallback query ke database PostgreSQL untuk memeriksa status sesi asli. Hasilnya kemudian disimpan kembali di Redis untuk mempercepat request berikutnya.
  5. PermissionsGuard kemudian membandingkan permissions dari perannya dengan perizinan yang dibutuhkan rute (`@Permissions()`).

Role-Based Access Control (RBAC) & Hak Akses

Starterkit ini mengimplementasikan otorisasi tingkat lanjut berbasis izin (permissions) yang dikelompokkan ke dalam peran (roles). Izin user dievaluasi secara dinamis oleh PermissionsGuard.

1. Matriks Peran & Hak Akses Default

Berikut adalah pemetaan peran default dan izin yang diberikan berdasarkan seeder:

Peran (Role) Deskripsi Izin (Permissions) Terkait
super_admin Akses penuh tanpa batas ke semua fitur sistem Semua izin (users.*, roles.*, permissions.*, audit_logs.read)
admin Akses manajemen sistem terpadu users.read, users.create, users.update, roles.read, permissions.read, audit_logs.read
user Pengguna umum sistem users.read (Hanya dapat membaca informasi profil sendiri/publik)

2. Penggunaan Decorator @Permissions

Proteksi rute controller Anda dengan menyematkan decorator @Permissions() bersama dengan guard otorisasi:

@Controller('roles')
@UseGuards(JwtAuthGuard, PermissionsGuard)
export class RolesController {
  @Post()
  @Permissions('roles.create')
  async create(@Body() dto: CreateRoleDto) {
    // Hanya user yang memiliki izin 'roles.create' yang dapat mengakses
  }
}

3. Cara Menambah Permission Baru ke dalam Sistem

Jika Anda menambah fitur baru yang memerlukan hak akses spesifik, ikuti langkah berikut:

  1. Buka file prisma/seed.ts.
  2. Tambahkan permission baru Anda (contoh: products.delete) ke dalam array basePermissions.
  3. Petakan permission baru tersebut ke dalam peran (role) yang sesuai di objek roles.
  4. Jalankan perintah seeder untuk menyinkronkan data database:
pnpm run db:seed

Cara Menambah Modul Baru

Gunakan langkah-langkah terstruktur berikut saat menambahkan fitur atau domain bisnis baru ke dalam starter kit:

Langkah 1: Definisikan Schema Database

Buka berkas prisma/schema.prisma, buat model baru dengan relasi yang dibutuhkan, serta tambahkan indeks pada kolom filter:

prisma/schema.prisma 
model Product {
  id          String   @id @default(uuid())
  name        String
  price       Float
  userId      String
  createdAt   DateTime @default(now())
  updatedAt   DateTime @updatedAt
  deletedAt   DateTime? // Soft-delete column

  @@index([userId])
  @@index([deletedAt])
  @@map("products")
}

Jalankan migrasi untuk memperbarui skema di database lokal Anda:

npx prisma migrate dev --name create_products_table

Langkah 2: Generate Modul Menggunakan Nest CLI

Gunakan NestJS generator CLI untuk membuat berkas modul secara rapi:

npx nest g module modules/products
npx nest g controller modules/products --no-spec
npx nest g service modules/products

Langkah 3: Tulis DTO & Lakukan Validasi Input

Buat DTO di dalam folder modules/products/dto menggunakan validator decorator:

modules/products/dto/create-product.dto.ts 
import { IsString, IsNumber, MinLength, IsPositive } from 'class-validator';

export class CreateProductDto {
  @IsString()
  @MinLength(3)
  name: string;

  @IsNumber()
  @IsPositive()
  price: number;
}

Langkah 4: Tulis Logika Bisnis & Proteksi Route

Implementasikan endpoint di Controller dengan proteksi Guard JWT, otorisasi izin spesifik, dan validasi kepemilikan data (anti-IDOR):

modules/products/products.controller.ts 
import { Controller, Post, Body, UseGuards } from '@nestjs/common';
import { JwtAuthGuard } from '../auth/guards/jwt-auth.guard';
import { PermissionsGuard } from '../../common/guards/permissions.guard';
import { Permissions } from '../../common/decorators/permissions.decorator';
import { CurrentUser } from '../../common/decorators/current-user.decorator';
import { CreateProductDto } from './dto/create-product.dto';
import { ProductsService } from './products.service';

@Controller('products')
@UseGuards(JwtAuthGuard, PermissionsGuard)
export class ProductsController {
  constructor(private readonly productsService: ProductsService) {}

  @Post()
  @Permissions('products.create')
  async create(
    @CurrentUser() user: any,
    @Body() dto: CreateProductDto
  ) {
    return this.productsService.create(user.id, dto);
  }
}

Langkah 5: Tulis Unit Test

Buat berkas *.spec.ts pendamping untuk menguji logika bisnis secara terisolasi dengan mem-mock PrismaService:

modules/products/products.service.spec.ts 
import { Test, TestingModule } from '@nestjs/testing';
import { ProductsService } from './products.service';
import { PrismaService } from '../../database/prisma.service';

describe('ProductsService', () => {
  let service: ProductsService;
  let prisma: any;

  beforeEach(async () => {
    prisma = {
      product: { create: jest.fn() }
    };

    const module: TestingModule = await Test.createTestingModule({
      providers: [
        ProductsService,
        { provide: PrismaService, useValue: prisma }
      ],
    }).compile();

    service = module.get(ProductsService);
  });

  it('should create product successfully', async () => {
    prisma.product.create.mockResolvedValue({ id: 'prod-1', name: 'Mug' });
    const res = await service.create('usr-1', { name: 'Mug', price: 10000 });
    expect(res.id).toBe('prod-1');
  });
});

Modul Enterprise Tambahan

Starterkit ini menyertakan 6 modul tingkat enterprise untuk mendukung kebutuhan aplikasi skala produksi tingkat lanjut:

1. OAuth2 Social Auth

Menangani login eksternal via Google dan GitHub dengan mekanisme auto-link akun dan auto-registrasi aktif secara aman.

  • Endpoint Login: GET /auth/oauth/google dan GET /auth/oauth/github
  • Endpoint Callback: GET /auth/oauth/google/callback dan GET /auth/oauth/github/callback (mengeluarkan HttpOnly cookie & redirect ke frontend web URL)

2. Outbound Webhooks & Event Dispatcher

Memungkinkan pengiriman event data sistem (seperti user.created) ke server eksternal milik klien Anda secara asinkron menggunakan BullMQ worker.

  • CRUD Subskripsi: /webhooks/subscriptions
  • Fitur Keamanan: Setiap payload webhook di-sign menggunakan HMAC SHA-256 dan dikirim dengan custom header X-Webhook-Signature untuk validasi keaslian.
  • Observability: Riwayat pengiriman tercatat lengkap di tabel WebhookDeliveryLog.

3. Billing & Payments Boilerplate

Integrasi dasar dengan **Stripe** dan **Midtrans Snap** untuk pembuatan checkout session/token pembayaran dan penanganan callback webhook pembayaran secara stateless.

  • Checkout API: POST /billing/checkout
  • Webhook Public Handlers: POST /billing/webhooks/stripe dan POST /billing/webhooks/midtrans
  • Integrasi otomatis dengan model Subscription pengguna untuk melacak masa aktif langganan.

4. Multi-Channel Notification Engine

Sistem notifikasi terpusat untuk mendistribusikan pesan via WebSocket (Socket.io) real-time, Email (nodemailer EJS template), dan Push Notification (FCM).

  • WebSocket Gateway: namespace /notifications terautentikasi JWT token.
  • Penyimpanan Token Perangkat: POST /notifications/devices
  • CRUD Notifikasi: GET /notifications (dengan paginasi) dan PATCH /notifications/:id/read

5. Developer API Key Management

Sistem pengelolaan API Key bagi developer pihak ketiga yang terintegrasi dengan pengamanan hash SHA-256 (hanya ditampilkan sekali saat pembuatan) dan pembatasan request dinamis menggunakan Redis cache.

  • CRUD API Keys: /api-keys
  • API Key Guard: Menyediakan ApiKeyGuard yang memverifikasi header X-API-Key, mencocokkan scope izin, serta melakukan rate limiting dinamis berbasis Redis per API Key.

6. Async Excel/CSV Export

Sistem ekspor data besar (misal: Daftar Pengguna atau Audit Log) secara asinkron. Mengirim tugas ke BullMQ worker, menyusun berkas Excel menggunakan exceljs, mengunggah ke S3, dan menotifikasi pengguna secara real-time via WebSocket saat berkas siap diunduh.

  • Trigger Export: POST /file-jobs/export
  • Status & Unduhan: GET /file-jobs dan GET /file-jobs/:id (mengembalikan link file S3 yang aman)

Integrasi Antrean & Pekerjaan Latar Belakang (BullMQ)

Starterkit telah mengonfigurasi BullMQ secara global untuk mengorkestrasi pekerjaan asinkronus (background jobs) yang memakan waktu lama, seperti pengiriman email atau konversi media.

1. Mendaftarkan Antrean di Fitur Modul

Mendaftarkan antrean baru bernama notifications ke dalam modul Anda:

modules/notification/notification.module.ts 
import { Module } from '@nestjs/common';
import { BullModule } from '@nestjs/bullmq';
import { NotificationService } from './notification.service';
import { NotificationProcessor } from './notification.processor';

@Module({
  imports: [
    BullModule.registerQueue({
      name: 'notifications',
    }),
  ],
  providers: [NotificationService, NotificationProcessor],
  exports: [NotificationService],
})
export class NotificationModule {}

2. Memproduksi Pekerjaan (Producer)

Suntikkan antrean ke dalam service Anda untuk menambahkan pekerjaan asinkronus baru:

modules/notification/notification.service.ts 
import { Injectable } from '@nestjs/common';
import { InjectQueue } from '@nestjs/bullmq';
import { Queue } from 'bullmq';

@Injectable()
export class NotificationService {
  constructor(
    @InjectQueue('notifications') private readonly notificationQueue: Queue,
  ) {}

  async sendWelcomeEmail(userId: string, email: string) {
    // Menambahkan job asinkronus ke Redis
    await this.notificationQueue.add('sendWelcome', {
      userId,
      email,
    }, {
      attempts: 3, // Coba ulang maksimal 3 kali jika gagal
      backoff: {
        type: 'exponential',
        delay: 5000, // Eksponensial delay mulai dari 5 detik
      }
    });
  }
}

3. Memproses Pekerjaan (Processor)

Buat class processor untuk mengonsumsi pekerjaan secara asinkronus di latar belakang:

modules/notification/notification.processor.ts 
import { Processor, WorkerHost } from '@nestjs/bullmq';
import { Job } from 'bullmq';
import { Logger } from '@nestjs/common';

@Processor('notifications')
export class NotificationProcessor extends WorkerHost {
  private readonly logger = new Logger(NotificationProcessor.name);

  async process(job: Job): Promise {
    switch (job.name) {
      case 'sendWelcome':
        this.logger.log(`Memproses welcome email untuk user ID: ${job.data.userId}`);
        // Logika pengiriman email menggunakan SMTP / Mailgun / AWS SES
        break;
      default:
        this.logger.warn(`Job tidak terdaftar: ${job.name}`);
    }
  }
}

4. Otorisasi & Monitoring Bull Board

Dashboard Bull Board dapat diakses pada endpoint /queues. Untuk melindunginya dari akses publik di produksi, kami menggunakan middleware QueuesAuthMiddleware:

  • Middleware ini memverifikasi JWT access token pengguna secara langsung dari cookies atau authorization header.
  • Hanya pengguna dengan role super_admin yang diizinkan untuk melihat dan mengelola antrean (seperti melakukan restart job yang gagal).

Transaksi Database & Zero-Downtime Migrations

Bagian ini membahas integritas transaksi database tingkat lanjut dan penanganan migrasi skema yang aman di lingkungan produksi.

1. Menggunakan Transaksi Database (ACID)

Setiap mutasi database yang saling berkaitan wajib dibungkus dalam transaksi untuk menghindari inkonsistensi data:

🔗 Transaksi Sekuensial (Sequential Transaction)

Digunakan bila kueri berikutnya membutuhkan data hasil keluaran kueri sebelumnya:

await prisma.$transaction(async (tx) => {
  const user = await tx.user.create({ data: { ... } });
  await tx.profile.create({ data: { userId: user.id } });
});

⚡ Transaksi Paralel (Parallel Transaction)

Digunakan bila kueri tidak saling bergantung, mengeksekusi array operasi sekaligus:

await prisma.$transaction([
  prisma.user.update({ ... }),
  prisma.session.updateMany({ ... })
]);

2. Panduan Zero-Downtime Migrations (Expand and Contract)

Untuk mencegah kegagalan runtime saat rilis bertahap (Rolling Update / Blue-Green), dilarang melakukan migrasi *breaking-changes* secara langsung. Terapkan pola 3-fase:

  1. Fase 1: Expand (Penyediaan Kolom Baru):
    • Tambahkan kolom baru di schema.prisma (wajib bersifat opsional / *nullable*).
    • Jalankan kueri dual-write di kode aplikasi (menulis data ke kolom lama dan baru secara bersamaan). Rilis kode ini ke produksi.
  2. Fase 2: Transition (Migrasi Data Historis):
    • Jalankan skrip latar belakang untuk menyalin data dari kolom lama ke kolom baru.
    • Ubah kueri pembacaan kode aplikasi agar mengambil data hanya dari kolom baru. Rilis kode ini.
  3. Fase 3: Contract (Pembersihan & Penghapusan):
    • Hapus kolom lama dari berkas skema dan jalankan migrasi untuk men-drop kolom lama di DB.
    • Hapus logika dual-write dari kode aplikasi. Rilis kode final ini.

Strategi File Upload Aman

Starterkit menerapkan mekanisme upload stateless langsung ke Object Storage S3 untuk menjamin skalabilitas horizontal dan keamanan berkas.

1. Validasi Magic Bytes (File Signature)

Jangan pernah mempercayai tipe MIME yang dideklarasikan oleh header HTTP klien karena mudah dipalsukan. Backend membaca data biner awal berkas (*magic bytes*) untuk mendeteksi tipe file sesungguhnya:

// Dynamic import dari library 'file-type'
const { fileTypeFromBuffer } = await import('file-type');
const fileType = await fileTypeFromBuffer(fileBuffer);

if (!allowedMimeTypes.includes(fileType.mime)) {
  throw new BadRequestException('MIME type tidak sah!');
}

2. Keamanan Akses Berkas dengan Pre-signed URLs

Berkas privat (seperti dokumen KTP, slip pembayaran, berkas medis) tidak boleh diakses secara langsung tanpa autentikasi. Gunakan token presigned URL dengan masa kedaluwarsa singkat:

import { GetObjectCommand } from '@aws-sdk/client-s3';
import { getSignedUrl } from '@aws-sdk/s3-request-presigner';

const command = new GetObjectCommand({
  Bucket: 'my-bucket',
  Key: 'uploads/private-file.pdf',
});

// Menghasilkan URL yang berlaku terbatas selama 60 menit
const url = await getSignedUrl(s3Client, command, { expiresIn: 3600 });

3. Validasi Ukuran & Tipe Berkas via Pipes

Untuk membatasi ukuran file dan format biner secara aman di controller endpoint, gunakan kombinasi ParseFilePipe:

@Post('upload')
@UseInterceptors(FileInterceptor('file'))
async uploadFile(
  @UploadedFile(
    new ParseFilePipe({
      validators: [
        new MaxFileSizeValidator({ maxSize: 1024 * 1024 * 5 }), // Max 5MB
        new FileTypeValidator({ fileType: 'image/(jpeg|png)' }),
      ],
    }),
  )
  file: Express.Multer.File,
) {
  return this.uploadService.uploadFile(file.buffer, file.originalname, file.mimetype);
}

Security, Rate Limiting & Caching

Proyek ini mengimplementasikan mitigasi celah keamanan performa tingkat lanjut:

⚡ Mitigasi CPU DoS (O(1) Refresh Token)

Dengan memformat refresh token menjadi sessionId.tokenRandomBytes, backend melakukan lookup berbasis indexed primary key terlebih dahulu secara langsung. Operasi hash Argon2 yang berat hanya dieksekusi satu kali pada session yang ditemukan, mencegah eksploitasi beban CPU.

1. Rate Limiting (Throttling)

ThrottlerModule terdaftar secara global membatasi request maksimal sebanyak 100 request dalam 1 menit. Anda dapat mengabaikan atau menimpa batasan ini di level controller/route tertentu:

Kustomisasi Throttler 
import { Throttle, SkipThrottle } from '@nestjs/throttler';

// Mengabaikan rate limiting untuk seluruh rute di Controller ini
@SkipThrottle()
@Controller('public')
export class PublicController {}

// Menimpa limit (maksimal 5 request per 10 detik) untuk rute login
@Throttle({ default: { limit: 5, ttl: 10000 } })
@Post('login')
async login() {}

2. Mekanisme Caching Redis

Data berikut disimpan di cache Redis guna menghindari overhead query database SQL di setiap request:

  • Validasi Session User: Disimpan pada key session:{sessionId}:user (TTL 5 menit). Di-invalidasi instan saat logout, ganti password, perubahan status user, dan penghapusan akun.
  • Izin User (Permissions): Disimpan pada key user:{userId}:permissions (TTL 1 jam). Di-invalidasi instan saat admin mengubah hak akses peran di panel roles.

3. Praktek Pembersihan Cache (Cache Invalidation Flow)

Untuk menjamin konsistensi data saat ada mutasi (perubahan data), ikuti alur *Cache Invalidation* berikut:

// 1. Lakukan update data di database
await this.prisma.user.update({
  where: { id: userId },
  data: { status: 'DELETED' }
});

// 2. Hapus cache terkait agar user ter-logout seketika
await Promise.all([
  this.cacheService.del(CacheKeys.sessionUser(sessionId)),
  this.cacheService.del(CacheKeys.userPermissions(userId))
]);

Standardisasi Error & Serialisasi Output

Mekanisme penyeragaman respons JSON di seluruh sistem sangat penting untuk konsistensi integrasi dengan aplikasi frontend/klien.

1. Format Error Terstandarisasi

Semua exception yang ditangkap di filter global akan diubah menjadi format standar:

{
  "success": false,
  "data": null,
  "error": {
    "code": "BAD_REQUEST",
    "message": "Nama minimal terdiri dari 3 karakter",
    "details": { ... } // Berisi detail validasi class-validator (jika ada)
  }
}

2. Pemetaan Kode Error Domain

Status HTTP Kode Error Domain Keterangan Kasus
400 Bad Request BAD_REQUEST Kegagalan validasi payload input atau format kueri
401 Unauthorized UNAUTHORIZED Token kedaluwarsa, tidak ada token, atau session tidak valid
403 Forbidden FORBIDDEN User tidak memiliki izin RBAC untuk mengakses rute terkait
404 Not Found NOT_FOUND Entitas database atau rute endpoint tidak ditemukan
409 Conflict CONFLICT Duplikasi data unik (misal: email ganda) atau data bentrok
429 Too Many Requests RATE_LIMITED Permintaan melebihi batas throttler limit
500 Internal Error INTERNAL_SERVER_ERROR Error tidak ditangani (Stack trace disembunyikan di production)

3. Serialisasi Data Sensitif

Jangan pernah melempar password hash ke luar secara tidak sengaja. Tandai field sensitif pada Class DTO atau Entity Anda dengan decorator `@Exclude()` dari class-transformer:

import { Exclude } from 'class-transformer';

export class UserEntity {
  id: string;
  name: string;
  email: string;

  @Exclude() // Otomatis disensor oleh ClassSerializerInterceptor
  passwordHash: string;
}

Testing, Linting & Observability

Menulis tes otomatis dan menjaga format kode adalah kewajiban dasar untuk menjaga stabilitas starter kit.

1. Testing Commands

🧪 Unit Testing

Meningkatkan keandalan fungsi di AuthService, UsersService, dan UploadService secara terisolasi tanpa efek samping database asli.

npm run test

🚀 End-to-End (E2E) Testing

Pengujian integrasi alur HTTP request penuh menggunakan database sandbox untuk memastikan keutuhan response data.

npm run test:e2e

2. Linting & Formatting

Selalu bersihkan dan format kode Anda sebelum melakukan git commit atau membuat pull request:

# Memeriksa dan memperbaiki linting otomatis (ESLint)
npm run lint

# Memformat pemformatan kode (Prettier)
npm run format

3. Observability & Health Check

Untuk menunjang liveness/readiness probe (Kubernetes/Docker), gunakan rute berikut:

  • Health Check Endpoint: GET /health (Memantau kesehatan PostgreSQL, konsumsi RAM heap memory, dan sisa kapasitas penyimpanan disk).
  • Correlation / Request ID: Setiap HTTP response melampirkan header x-request-id yang di-generate baru secara dinamis per request untuk memudahkan pelacakan log secara terpusat di sisi production server.
